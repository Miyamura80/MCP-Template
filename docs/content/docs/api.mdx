## Base URL

```
http://localhost:8000
```

## Authentication

Include your API key in the `Authorization` header:

```bash
curl -H "Authorization: Bearer sk_live_..." \
  http://localhost:8000/api/v1/auth/me
```

Enterprise SSO via WorkOS is also supported.

## Calling Tools

All tools are available as POST endpoints:

```
POST /api/v1/services/{tool_name}
```

### Examples

**Greet a user:**

```bash
curl -X POST http://localhost:8000/api/v1/services/greet \
  -H "Authorization: Bearer sk_live_..." \
  -H "Content-Type: application/json" \
  -d '{"name": "Alice", "shout": true}'
```

```json
{ "message": "HELLO, ALICE!", "times": 1 }
```

**Get a config value:**

```bash
curl -X POST http://localhost:8000/api/v1/services/config_get \
  -H "Authorization: Bearer sk_live_..." \
  -H "Content-Type: application/json" \
  -d '{"key": "llm_config.cache_enabled"}'
```

```json
{ "key": "llm_config.cache_enabled", "value": true }
```

**Run health checks:**

```bash
curl -X POST http://localhost:8000/api/v1/services/doctor \
  -H "Authorization: Bearer sk_live_..." \
  -H "Content-Type: application/json" \
  -d '{"fix": false}'
```

### All Available Tools

| Endpoint | Body | Response |
|----------|------|----------|
| `POST /api/v1/services/greet` | `{"name": "...", "shout": false, "times": 1}` | `{"message": "...", "times": 1}` |
| `POST /api/v1/services/config_show` | `{}` | `{"config": {...}}` |
| `POST /api/v1/services/config_get` | `{"key": "..."}` | `{"key": "...", "value": ...}` |
| `POST /api/v1/services/config_set` | `{"key": "...", "value": "..."}` | `{"key": "...", "coerced_value": ...}` |
| `POST /api/v1/services/doctor` | `{"fix": false}` | `{"checks": [...], "has_failures": false}` |

## Health Check

No authentication required.

```bash
curl http://localhost:8000/health
```

```json
{
  "status": "ok",
  "version": "0.1.1",
  "commit": "f7be131",
  "timestamp": "2025-01-15T10:30:00+00:00",
  "components": {
    "api": { "status": "ok" },
    "database": { "status": "ok" },
    "redis": { "status": "not_configured" },
    "stripe": { "status": "ok" }
  }
}
```

Status is `"ok"` when all components are healthy, `"degraded"` if any have errors.

## API Key Management

### Create a key

```bash
curl -X POST http://localhost:8000/api/v1/auth/api-keys \
  -H "Authorization: Bearer sk_live_..." \
  -H "Content-Type: application/json" \
  -d '{"name": "my-integration", "scope_template": "standard", "expires_in_days": 90}'
```

**Scope templates:**

| Template | What it allows |
|----------|---------------|
| `read_only` | Read config and billing info |
| `standard` | Read + execute tools + manage billing |
| `admin` | Everything |

Or provide explicit scopes:

```json
{ "name": "limited-key", "scopes": ["services:read", "services:execute"] }
```

### List your keys

```bash
curl -H "Authorization: Bearer sk_live_..." \
  http://localhost:8000/api/v1/auth/api-keys
```

### Revoke a key

```bash
curl -X DELETE -H "Authorization: Bearer sk_live_..." \
  http://localhost:8000/api/v1/auth/api-keys/{key_id}
```

## Scopes

Each API key has scopes that control what it can do:

| Scope | Description |
|-------|-------------|
| `services:read` | Read tool metadata |
| `services:execute` | Call tool endpoints |
| `billing:read` | View usage and subscription info |
| `billing:write` | Report usage |
| `admin:read` | Read admin data |
| `admin:write` | Admin operations |

Wildcards: `*` grants all scopes, `services:*` grants all service scopes.

## Billing & Usage

### Check your usage

```bash
curl -H "Authorization: Bearer sk_live_..." \
  http://localhost:8000/api/v1/billing/usage/current
```

### Upgrade your plan

```bash
curl -X POST -H "Authorization: Bearer sk_live_..." \
  http://localhost:8000/api/v1/billing/checkout
```

Returns a Stripe checkout URL to complete the upgrade.

### Check subscription status

```bash
curl -H "Authorization: Bearer sk_live_..." \
  http://localhost:8000/api/v1/billing/subscriptions/{subscription_id}
```

## Rate Limits & Quotas

- Requests are rate-limited per client
- Each subscription tier has a daily quota (e.g. free: 100 requests/day, pro: 10,000/day)
- When you exceed your daily quota, the API returns `402` with a `Retry-After` header indicating when the quota resets (next day)

## Errors

All errors return a JSON object:

```json
{ "detail": "Error message describing what went wrong" }
```

| Code | Meaning |
|------|---------|
| `400` | Invalid input |
| `401` | Missing or invalid API key |
| `402` | Quota exceeded |
| `403` | Insufficient permissions (check your key's scopes) |
| `404` | Not found |
| `429` | Rate limited (try again later) |
